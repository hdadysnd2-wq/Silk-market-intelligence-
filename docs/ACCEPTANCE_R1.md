# قائمة قبول R1 — البرنامج الشامل (A→F + تمرير النثر) على تشغيلة حيّة

**الحالة المرجعية:** `main @ f40463d` — يضمّ PR-A→PR-F (البرنامج الشامل)
+ PR-C0 (تمرير النثر). ٨٨٠ اختباراً هرمِتياً أخضر.

**حالة القبول:** **هولندا × تمور (HS 080410)** — تطابق العيّنة المحفوظة
(`samples/client_report_latest.docx`) فتكون المقارنة مباشرة، وتُفعّل كل
موجة: بوابة الأهلية الأوروبية (EU 2017/625)، إشارات Eurostat، الشريحة
الحلال، والموقع السعري السعودي.

**السكربت المرافق:** `tools/acceptance_run.py` — يشغّل كل ما تحته آلياً
ويحفظ المخرجات في `acceptance_out/<الطابع الزمني>/`. الصقْ مفتاحك وشغّل:

```bash
SILK_API_BASE="https://<your-railway-app>"  SILK_API_KEY="<your-key>" \
  python3 tools/acceptance_run.py --product "تمور" --market "Netherlands" --hs 080410
```

> **تنبيه اعتمادات:** تشغيلة `/research` كاملة تكلّف ~$1 (أرقام Railway حيّة).
> شغّل مرة واحدة؛ إن تعثّرت استخدم `resume` لا إعادة الإطلاق (حرق مضاعف).
> اللقطة السريعة والدردشة تحجزان تفعيلة كلود واحدة لكلٍّ من السقف اليومي.

## ٢. الدردشة المؤرَّضة (grounded chat) — ثلاثة أسئلة

عبر `POST /analyses/{id}/ask` بجسم `{"question": "…"}`. الأسئلة الثلاثة:

1. «ما حجم واردات هولندا من التمور وكم نموّها؟» (رقم من الدراسة)
2. «مَن أبرز المنتجات/العلامات المنافسة وبأي أسعار؟» (جدول المنافسة)
3. «ما بوابة الأهلية التنظيمية الأولى للدخول؟» (اشتراط EU 2017/625)
4. سؤال ضبط سلبي: «ما عدد سكان اليابان؟» (خارج الدراسة عمداً)

| # | المعيار | نجاح | فشل |
|---|---|---|---|
| 2.1 | **الإرضاء ببيانات الدراسة فقط** | إجابات ١-٣ تقتبس أرقام/حقائق ظهرت فعلاً في هذه الدراسة (قابلة للمطابقة مع docx) | إجابة تأتي برقم لا وجود له في الدراسة (اختلاق) |
| 2.2 | **إعلان الغياب لا الاختلاق** | سؤال الضبط (٤) يعيد «غير متوفر في هذا التحليل» أو ما يعادله، لا رقماً مختلَقاً | يجيب برقم عام من خارج الدراسة |
| 2.3 | **بلا تسريب تِلِمِتري** | لا اسم أداة/بعثة/رمز حكم في الإجابة | تسريب مصطلح داخلي |
| 2.4 | **التكلفة والسقف** | كل سؤال = تفعيلة كلود واحدة محتسَبة؛ عند النفاد يتدهور معلناً (`note`) لا 429 | 429 على مسار الدردشة، أو تجاوز صامت للسقف |

## ٧. مسطرة جاهزية العميل — عشرة معايير على التقرير الحيّ

سجّل كل معيار **٠ / ١** (٠=غائب/مكسور، ١=مستوفى). **عتبة القبول: ≥ ٨/١٠**،
وأي صفر في المعايير المميّزة بـ **(حاسم)** يُسقِط القبول مهما كان المجموع.

| # | المعيار | ١ يُمنَح حين |
|---|---|---|
| R1 | **أطروحة قرار** (حاسم) | الخلاصة تذكر «التوصية X لأن…؛ وتتحوّل إلى GO إذا…» لا وصفاً محايداً |
| R2 | **حكم واحد متّسق** (حاسم) | حكم توليف واحد فقط، متطابق عبر docx/مختصر/لوحة — لا حكمان |
| R3 | **صفر اختلاق** (حاسم) | كل رقم مصدره مذكور؛ الفجوات «غير مرصود»؛ صفر أرقام بلا سند |
| R4 | **صفر تسريب للعميل** (حاسم) | مسح المصطلحات المحظورة على docx العميل نظيف (البند 1.2) |
| R5 | **جدول منافسة حيّ** | ≥٣ منتجات بأسعار مرصودة فعلية + سطر كومتريد المرجعي (البند 1.3) |
| R6 | **حجم الفرصة محسوب** | TAM/SAM/SOM في جدول بأساس كل رقم، ثم تفسيره نثراً |
| R7 | **عمق ثقافة المستهلك** | استعلامات مرتبطة/صاعدة + مواسم + بُعد ديني، لا رقم اهتمام واحد |
| R8 | **مسار دخول عملي** | خارطة ٩٠ يوماً + القناة الأولى الموصى بها بسببها + بوابة الأهلية أولاً |
| R9 | **كل قسم يُترجَم لقرار** | كل قسم رئيسي يُختَم بـ«ماذا يعني هذا لقرارك:» |
| R10 | **بوابة الجودة** | `quality_gate.verdict` = PASS أو PASS-WITH-WARNINGS (لا FAIL) |

**المخرَج:** المجموع _/١٠ + حالة المعايير الحاسمة + قرار القبول (اقبل /
اقبل بتحفّظات / أعد التشغيل).

## قرار القبول النهائي

- **اقبل** — ≥٨/١٠، كل الحاسمة ١، البنود ١-٦ ضمن عتباتها.
- **اقبل بتحفّظات** — ٨/١٠ مع ملاحظات غير حاسمة موثَّقة (مثلاً تغطية ٧/١٢
  بسبب مفتاح مصدر مفقود لا خلل منطقي).
- **أعد التشغيل / أوقف** — أي معيار حاسم = ٠، أو تغطية <٨/١٢، أو FAIL من
  بوابة الجودة، أو تسريب مصطلح محظور على docx العميل.

> بعد التشغيلة: سجّل النتيجة في `PATCH /analyses/{id}/outcome` (السجل
> التراكمي)، واحفظ مجلّد `acceptance_out/<الطابع>/` كأثر مراجعة.
