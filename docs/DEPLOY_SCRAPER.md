# نشر مكشطة خرائط قوقل — Google Maps scraper (C1, Command #5a)

> **قرار D-03 (سجلّ القرارات):** كلود يكتب خطوات كونسول Railway ويتوقّف.
> الأوامر C2–C5 (تكامل الكشط) **لا تُفتَح** إلا بعد أن يؤكّد المالك أن الخدمة
> الثانية حيّة. كلود لا يستطيع تزويد Railway — فأمرٌ لا يُغلق نفسه ليس أمراً.
>
> **Owner action required.** This file is the exact console runbook. Do the
> steps, set `SILK_GMAPS_SCRAPER_URL`, redeploy the main app, then confirm the
> scraper service is live so Command #5b can open.

---

## لماذا خدمة ثانية منفصلة (عزل المخاطرة)

المكشطة تُشغِّل متصفّحاً حقيقياً ضدّ خرائط قوقل — قد يحجبها قوقل أو تتعطّل.
لذلك **لا تُضمَّن في حاوية بايثون**: تُنشَر خدمة Railway مستقلّة على **الشبكة
الخاصة فقط** (never public). إن سقطت، التطبيق الرئيس + القواعد + المهام
+ `/health` تبقى سليمة تماماً (تعطيل نظيف عبر `SILK_GMAPS_SCRAPER_URL`).

الصورة الرسمية: `gosom/google-maps-scraper` بوضع **خادم الويب + REST API**.

---

## الخطوات الدقيقة في كونسول Railway

### ١) أنشئ الخدمة الثانية من الصورة الرسمية
1. في **نفس مشروع Railway** الذي يحوي التطبيق الرئيس: **New → Service →
   Deploy a Docker Image**.
2. الصورة: `gosom/google-maps-scraper:latest`
   (المستودع الرسمي: `github.com/gosom/google-maps-scraper`).
3. سمِّ الخدمة `gmaps-scraper` (يحدّد اسم مضيفها الداخلي لاحقاً).
4. **Deploy Command / Args** — شغّلها بوضع خادم الويب لا وضع CLI:
   `-web -addr :8080 -data-folder /data` (يفتح REST API على المنفذ 8080).
   > راجع README الصورة إن تغيّرت الأعلام؛ المطلوب: web-server mode + REST.

### ٢) أعطها وحدة تخزين خاصة بها (data volume)
1. من تبويب الخدمة: **Settings → Volumes → New Volume**.
2. Mount path: `/data` (يطابق `-data-folder /data` أعلاه).
3. هذه الوحدة للمكشطة وحدها — **منفصلة تماماً** عن وحدة `/data` للتطبيق الرئيس
   (وحدة Railway تُركَّب على خدمة واحدة فقط؛ لا تشاركها).

### ٣) الشبكة الخاصة فقط — لا تعريض عام
1. **Settings → Networking**: **لا تفعّل «Generate Domain»** ولا أي منفذ عام.
2. اترك **Private Networking** مفعّلاً (الافتراضي داخل المشروع). عنوانها
   الداخلي سيكون بالشكل: `http://gmaps-scraper.railway.internal:8080`
   (استبدل الاسم إن سمّيتها غير `gmaps-scraper`).
3. تحقّق: زر «Public Networking» يجب أن يبقى مُطفأً — لا URL عام إطلاقاً.

### ٤) اربط التطبيق الرئيس بها (متغيّر واحد)
1. من خدمة **التطبيق الرئيس** (لا المكشطة): **Variables → New Variable**:
   ```
   SILK_GMAPS_SCRAPER_URL = http://gmaps-scraper.railway.internal:8080
   ```
2. **Redeploy** خدمة التطبيق الرئيس ليلتقط المتغيّر.

### ٥) أكّد أن الخدمة حيّة
1. `/health` للتطبيق الرئيس يجب أن يعرض:
   `"gmaps_scraper": "on — مُهيَّأة (شبكة خاصة، خدمة منفصلة)"`.
   ```bash
   curl -sS "$BASE/health" -H "X-API-Key: $KEY" | python3 -m json.tool | grep gmaps
   ```
2. لوجات خدمة `gmaps-scraper` تُظهِر خادم الويب استمع على `:8080`.

عند إتمام ٥ **بلّغ**: «الخدمة حيّة» — فيُفتَح Command #5b (تكامل C2–C5).

## التعطيل النظيف (clean-disable) — مضمون بالشيفرة

`silk_gmaps.py` هو واجهة التهيئة: `enabled()` تقرأ `SILK_GMAPS_SCRAPER_URL`
فقط. **غيابه = تعطيل كامل** — لا نداء يُحاوَل، و`/health` يعرض
`off — … (المكشطة معطّلة تعطيلاً نظيفاً — لا أثر على المهام أو جهوزية البحث)`.

- **إيقاف المكشطة لاحقاً:** احذف `SILK_GMAPS_SCRAPER_URL` من متغيّرات التطبيق
  الرئيس وأعِد النشر — أو أوقِف خدمة `gmaps-scraper`. في الحالتين، التطبيق
  الرئيس و`research_ready` و`/health` لا تتأثّر (`gmaps_scraper` حقل إخباري
  لا يحجب الجهوزية — مؤكَّد باختبار `tests/test_gmaps_scraper_c1.py`).
- **معيار القبول ٥ (FINAL ACCEPTANCE):** إيقاف خدمة المكشطة لا يؤثّر على
  `/health` الرئيس — هذا هو العزل الذي تضمنه هذه البنية.

> لا مفتاح مصدر يُضبَط في هذه الخطوات — فقط عنوان داخلي خاص. تبقى المفاتيح
> في متغيّرات Railway للتطبيق الرئيس كما هي (قاعدة أمن المصادر).

## واجهة REST المفترَضة + متغيّرات الضبط (C2–C5)

`silk_gmaps.py` يخاطب المكشطة على المسارات التالية (واجهة gosom web-mode):

| العملية | الطلب |
|---|---|
| تقديم مهمة | `POST {url}/api/v1/jobs` بجسم `{name, keywords[], depth, email, max_time}` → `{id}` |
| حالة/نتائج | `GET {url}/api/v1/jobs/{id}` → `{status, results}` (status «ok» = اكتملت) |
| تنزيل (إن لزم) | `GET {url}/api/v1/jobs/{id}/download` |

> **تحقّق عند أول تشغيل حيّ:** إن اختلفت مسارات نسختك من المكشطة، عدّل
> الثوابت في `silk_gmaps.submit_scrape`/`_fetch_job`. أي عدم تطابق يتدهور
> بأمان (كل نداء يفشل ⇒ السلسلة الاحتياطية Places ⇒ فجوة معلنة)، فلن يكسر
> التشغيلة — لكن المكشطة لن تُستعمَل حتى تُضبَط المسارات. راقب سطر السجلّ
> `gmaps leads path=…` لمعرفة أيّ مسار خدم فعلاً (scraper/places/cache/gap).

متغيّرات ضبط اختيارية (على التطبيق الرئيس):

| المتغيّر | الافتراضي | الأثر |
|---|---|---|
| `SILK_GMAPS_TIMEOUT_SECONDS` | 480 (٨ د) | المهلة الصلبة للكشط غير المتزامن (D-02) |
| `SILK_GMAPS_POLL_SECONDS` | 10 | فاصل الاستطلاع |
| `SILK_GMAPS_COLLECT_GRACE_S` | 30 | أقصى انتظار عند الجمع بعد الذيل (كي لا يزيد زمن التشغيلة) |

**معيار القبول (C2–C5):** جدول «قائمة مستوردين وموزعين قابلين للتواصل»
بجهات اتصال حقيقية + سطر `path=` مطبوع + إيقاف خدمة المكشطة لا يؤثّر على
`/health` الرئيس — الأخيران مقفولان باختبار؛ جهات الاتصال الحقيقية تظهر في
أول تشغيلة حيّة على النشر (المكشطة على الشبكة الخاصة غير قابلة للوصول من CI).
