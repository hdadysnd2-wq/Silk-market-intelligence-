# القسم الأول: محرّك التقاطع (Correlation Engine)

---

## ١. المشكلة التي تحلّها (جوهر القرار)

المنصة الحالية **تجميع (aggregation)**: كل وكيل يجيب عن سؤاله بمعزل،
وكلود يلخّص فوق الأكوام. الناتج: وصف للسوق ("السوق فيه منافسة،
الأسعار كذا").

المطلوب **ربط (correlation)**: البيانات تتقاطع حول منتج المستخدم
تحديداً لتنتج قراراً ("منافسك المباشر فلان، سعره كذا، هامشك ضده
كذا، بابك الواقعي للدخول هو القناة الفلانية، وأول مورّد تكلمه اسمه
كذا").

الفرق: تقرير عن السوق ← خطة دخولك أنت للسوق.

## ٢. تغيير المدخل — بطاقة المنتج (شرط مسبق)

بدون معرفة منتج المستخدم كطرف في المعادلة، أي ربط يظل نظرياً.
يُضاف لنموذج الإدخال حقل اختياري (لا إلزامي — التحليل الوصفي الحالي
يبقى متاحاً لمن لا يملك الأرقام):

```json
"product_card": {
  "cost_per_unit": 12.5,        // تكلفة الإنتاج/الشراء للوحدة (ريال أو $)
  "unit": "kg",
  "tier": "premium|standard|economy",  // فئة الجودة
  "monthly_capacity": 5000       // الطاقة الإنتاجية (اختياري)
}
```

- إن غاب `product_card`: المنصة تعمل كالحالي (تحليل وصفي) وتعرض
  بوضوح: "أضف بطاقة منتجك للحصول على تحليل تنافسي مخصص".
- إن وُجد: يتفعّل محرّك التقاطع ويُضاف قسم "موقعك التنافسي" للمخرجات.

## ٣. المعمارية — طبقة واحدة جديدة، لا هدم

```
الوكلاء الـ21 (كما هم، بلا تغيير)
        │  نتائجهم الخام الموسومة
        ▼
   correlation.py  ← الطبقة الجديدة الوحيدة
        │  خيوط مربوطة (correlated threads)
        ▼
   silk_synthesis (المرحلة ٢ تستقبل الخيوط بجانب الملخصات)
        ▼
   المخرجات (قسم جديد: "موقعك التنافسي")
```

**قاعدة صارمة:** `correlation.py` لا يستدعي أي API خارجي ولا كلود —
يعمل حصراً على نتائج الوكلاء الموجودة في الذاكرة. صفر تكلفة إضافية،
صفر سطح حقن جديد. أي ربط يحتاج بيانات غير موجودة يُعلن فجوة
("خيط غير مكتمل: سعر المنافس X غير مرصود") — نفس مبدأ "لا اختلاق".

## ٤. الخيوط الثلاثة (مخرجات المحرّك)

### خيط ١ — ملف المنافس المترابط (competitor_thread)

لكل منافس مُسمّى من `competitors_agent`:

- اسمه ← سعره إن رُصد في `retail_price` / `bestsellers` (مطابقة
  بالاسم، fuzzy match بحذر، وإلا "سعر غير مرصود")
- ← موقعه/مصنعه إن ظهر في `factories` / بيانات `/deepen` (Volza/Maps)
- ← قناته إن ظهرت في `distribution_channels` / `ecommerce_landscape`

```json
{"name": "الفوعة", "observed_price": {"value": 95, "unit": "SAR/kg",
 "source": "noon"}, "suppliers": [...], "channels": ["لولو", "كارفور"],
 "thread_completeness": "3/4"}
```

### خيط ٢ — الجدوى مقابل كل منافس (feasibility_thread)

لكل منافس له سعر مرصود:

- تكلفة المستخدم (من بطاقة المنتج)
- + الرسوم الجمركية (من `tariff`/WITS)
- + تقدير شحن معلَن كافتراض قابل للتعديل (لا رقم مخفي)
- = هامش المستخدم عند مضاهاة سعر المنافس، وعند البيع أقل منه 10%

**هذا الخيط هو حاسبة هامش الربح نفسها** (من قائمة V4 السابقة)
مربوطة بمنافسين حقيقيين بدل أرقام مجردة — لذلك تُحذف الحاسبة
كبند مستقل من V4 وتُدمج هنا.

### خيط ٣ — أبواب الدخول (entry_thread)

- القنوات المرصودة التي تبيع فئة المنتج (`tier` من بطاقة المنتج
  يطابَق مع فئة القناة إن رُصدت)
- مرتّبة: قناة يهيمن عليها منافس واحد بعقود = "صعبة"، قناة متعددة
  الموردين أو إلكترونية مفتوحة = "واقعية"
- المستوردون/الموردون النشطون فيها بالاسم (من طبقة `/deepen` إن
  فُعّلت، وإلا: "فعّل التعميق للحصول على الأسماء")

### خيط ٤ — جهات الاتصال الفعلية (contacts_thread) — قرار معتمد

**القرار:** بيانات explee (أسماء صنّاع القرار + إيميلات موثّقة)
تُدمج **داخل التقرير نفسه** كخاتمة عملية للتحليل — لا خطوة يدوية
خارج المنصة. التقرير ينتهي بـ"وهذا أول شخص تكلّمه" لا بـ"وهذه
خلاصة السوق".

**آلية التنفيذ (ضمن `/deepen` القائمة):**

- عند تفعيل التعميق على سوق GO، يُبنى استعلام explee **جراحي
  واحد آلياً** من مخرجات الخيوط السابقة: الفئة (من بطاقة المنتج)
  + السوق + نوع الجهة المستهدفة (من خيط أبواب الدخول) — بالإنجليزية
  (قاعدة explee عالمية، الاستعلام الإنجليزي أدق)
- النتيجة (١٥-٢٥ جهة: اسم، منصب، شركة، إيميل موثّق) تُخزَّن في
  SQLite (الهجرة إلى Postgres قرار مؤجل عند الحاجة) **مربوطة
  بذلك التحليل تحديداً** + تاريخ السحب +
  حقل صلاحية (٩٠ يوماً)
- تظهر في المخرجات: قسم "جهات التواصل المقترحة" بالتقرير الكامل
  واللوحة، وسطر واحد بالمختصر ("١٨ جهة اتصال موثّقة — التفاصيل
  باللوحة")
- بعد انقضاء الصلاحية: البيانات تُعرض بتحذير صريح "سُحبت قبل
  X يوماً — حدّثها قبل الاعتماد عليها" — لا تُخفى ولا تُعرض
  كأنها طازجة (نفس مبدأ الشفافية)

**حدود صريحة (مهمة قانونياً):**

- التخزين **مربوط بالتحليلات الفعلية فقط** — لا سحب شامل مسبق
  لأسواق لم تُحلَّل ("bulk pre-fetch" يخالف شروط خدمة أدوات
  البيانات ويبني قاعدة مشتقة محظورة، ويُدخل نطاق أنظمة حماية
  البيانات الشخصية بلا داعٍ)
- بيانات الاتصال بيانات أشخاص: تُستخدم لغرض التواصل التجاري
  المباشر المرتبط بالتحليل، ولا تُصدَّر جماعياً خارج هذا السياق
- الاشتراك المدفوع في explee يُفعَّل فقط بعد استهلاك الحصة
  المجانية (333 إيميلاً + 500 رصيد + 2000 صف إثراء) على أسواق
  GO حقيقية — والدفع بالاستهلاك (~50$/1000 جهة) لا اشتراك شهري
  ثابت [تصحيح للمواصفة السابقة التي وصفته اشتراكاً]

**التراكم الطبيعي:** بعد سنة استخدام، قاعدة البيانات تحوي جهات
اتصال كل الأسواق المدخولة فعلياً، موسومة بتواريخها وحالة تحليلها
ونتيجته (عمود outcome) — أصل تجاري حقيقي بُني بالاستخدام المشروع،
لا بالسحب الشامل.

### توسيع معتمد — explee كمغذٍّ شامل لبيانات الشركات (لا جهات الاتصال فقط)

**القرار:** بيانات explee المهيكلة عن الشركات (الحجم، إشارات
التوظيف، التمويل، النشاط) تُغذّي — عند تفعيل `/deepen` — ثلاثة
وكلاء قائمين إضافةً لخيط جهات الاتصال:

| الوكيل | ما يضيفه explee فوق مصدره الحالي |
|---|---|
| `competitors_agent` | ملف الشركة المنظّم: عدد الموظفين، هل توظّف الآن (توسّع/انكماش)، تمويلها — إشارات قوة المنافس الفعلية التي لا يوفرها بحث الويب |
| `factories_agent` | مصنّعون بفئة وحجم محددين كقائمة مهيكلة ("مصانع أغذية، ٥٠-٥٠٠ موظف") بدل نتائج بحث متناثرة |
| `importers_agent` | مستوردون/موزعون كملفات شركات كاملة — ترقية جوهرية لأضعف وكيل حالياً (best-effort ويب) |

**قواعد التنفيذ الصارمة:**

1. **حصراً ضمن `/deepen`** — المسار العادي لا يلمسه (نفس درس
   Volza التاريخي، لا استثناء)
2. **دفعة واحدة لا أربعة استعلامات:** استعلام explee واحد (أو
   اثنان كحد أقصى) عن السوق يُرجع دفعة شركات تتوزع على الوكلاء
   الثلاثة + خيط جهات الاتصال معاً — نفس الرصيد يعمل أربع مرات
3. **البيانات تُدمج موسومة بمصدرها:** كل حقل قادم من explee
   يظهر بوسم `source: explee` وتاريخ السحب — لا يُخلط بصمت مع
   نتائج بحث الويب المجاني (شفافية المصدر، كالعادة)
4. **نفس حقل الصلاحية (90 يوماً)** ينطبق على بيانات الشركات
   كما على جهات الاتصال

## ٥. تغيير دور كلود (المرحلة ٢ من synthesis)

البرومبت الحالي: "لخّص هذه البيانات وأصدر حكماً".

البرومبت الجديد (عند وجود بطاقة منتج): "أمامك مواجهة محددة —
منتج المستخدم (المواصفات المرفقة) ضد المنافسين التاليين (الخيوط
المرفقة). احكم على المواجهة: مَن يستطيع منافسته سعرياً ومن لا،
وما استراتيجية الدخول الواقعية عبر الأبواب المرصودة."

- الخيوط تُمرَّر داخل `raw_findings` بنفس حماية الحقن القائمة
  (لا تغيير في آلية العزل).
- الحكم الناتج يستشهد بالخيوط: "تستطيع منافسة X عند 78 ريال
  بهامش 25%" — كل رقم قابل للتتبع لخيطه.

## ٦. المخرجات — قسم جديد لا استبدال

يُضاف قسم "موقعك التنافسي" في:

- **اللوحة:** بطاقة أولى بعد الحكم مباشرة (قبل بطاقات المجموعات)
- **التقرير الكامل:** قسم بعد الخلاصة التنفيذية
- **المختصر:** سطران — أقرب منافس قابل للمنافسة + أفضل باب دخول

كل خيط غير مكتمل يظهر بشفافية: "3 من 5 منافسين لهم أسعار مرصودة —
لرفع التغطية فعّل طبقة التعميق."

## ٧. معايير القبول الخاصة بهذه المرحلة

1. تحليل بلا بطاقة منتج = سلوك المنصة الحالي بالضبط (لا انحدار)
2. تحليل ببطاقة منتج على سوق حقيقي: خيط منافس واحد على الأقل
   مكتمل (اسم + سعر مرصود) وهامش محسوب ظاهر بالمخرجات الثلاثة
3. اختبار "خيط ناقص": منافس بلا سعر مرصود يظهر بعلامة صريحة،
   لا يُخترع له سعر ولا يُسقط بصمت (نفس البروتوكول الرباعي، البند ٤)
4. اختبار حقن على الخيوط: اسم منافس يحوي نص حقن هجومي يُعزل
   في raw_findings كالمعتاد (نفس اختبار silk_synthesis القائم)
5. صفر استدعاءات API جديدة من correlation.py (يُثبت بعدّاد
   الاستدعاءات = 0 أثناء عمله)

## ٨. ما ليس ضمن النطاق (صراحةً)

- لا RAG ولا ذاكرة تراكمية (قرار مستقل سابق — تبقى مؤجلة)
- لا تسعير ديناميكي أو توصية سعر "مثالية" — المحرّك يحسب هوامش
  عند سيناريوهات معلنة، لا يدّعي معرفة السعر الأمثل
- لا مطابقة منتج-منافس بالذكاء الاصطناعي في هذه النسخة — fuzzy
  match نصي بسيط مع عتبة متحفظة، والغامض يُعلن غامضاً

# القسم الثاني: دروس "لو أُعيد البناء" — المعمارية الأنحف

> مصدر هذا القسم: تمرين فكري بُني على ست ثغرات جوهرية اكتُشفت
> وأُصلحت أثناء بناء V3 (استدعاء مدفوع تلقائي ×3، حذف صامت،
> ثقة مختلقة، طبقة حكم مزدوجة). **يُطبَّق بالدمج التدريجي ضمن V4
> وما بعدها، لا بإعادة بناء.**

## ٩.١ المبدأ الحاكم

المعمارية الأساسية (وكلاء يجمعون → كلود يحلل → مخرجات) صحيحة
ومُثبتة — لا تتغير. التغيير في **نقل الدروس من "قواعد نُراجعها"
إلى "بنية تفرضها"**.

## ٩.٢ ما يُدمج (من ٢١ وكيلاً إلى ~١٢)

أغلب الثغرات المكتشفة كانت في **الوصلات بين الوكلاء، لا داخلهم** —
وصلات أقل = سطح خطأ أقل:

| الوكلاء الحاليون | يصيرون | المبرر |
|---|---|---|
| cultural + business_culture + exhibitions | وكيل سلوك سوق واحد بثلاثة استعلامات | ثلاثتهم "بحث ويب عن سلوك السوق" — التقسيم إداري لا وظيفي |
| distribution_channels + ecommerce_landscape | وكيل قنوات واحد | نفس السؤال بقناتين (فعلي/رقمي) |
| religion_agent + currency_risk_agent | جداول lookup داخل وكيل الاقتصاد | بيانات شبه ثابتة (CSV مرجعي) لا تستحق "وكيلاً" |

**قاعدة التطبيق التدريجي:** لا يُدمج وكيلان إلا عند أول تعديل
جوهري يمسّهما أصلاً — الدمج يركب موجة صيانة قائمة، لا يفتح
جبهة جديدة.

## ٩.٣ ما يُحذف

- **طبقة `jury.ai` كلياً:** طبقة حكم ثانية موازية لـsynthesis —
  غرضان متداخلان، وكل إصلاح (حقن، ثقة) لزم تكراره مرتين.
  synthesis ثنائية المرحلة تكفي وحدها.
  **شرط الحذف:** بعد اكتمال محرّك التقاطع واستقراره فقط — لأن
  الحذف قبله يغيّر شكل الاستجابة التي تعتمد عليها الواجهة.

## ٩.٤ ما يُبنى مختلفاً — BaseAgent يفرض البروتوكول بنيوياً

**أهم قرار في هذا القسم.** البروتوكول الرباعي الحالي (تصنيف،
/analyze مقابل /deepen، اختبار انحدار، فشل مرئي) يعتمد على
الالتزام والمراجعة. البديل: فئة أساس `BaseAgent` **تفرضه بالوراثة**:

```python
class BaseAgent:
    PAID: bool = False          # تصنيف إلزامي عند التعريف
    def run(self, ...):
        if self.PAID and not deepen_context():
            return self._skipped("paid_agent_outside_deepen")
        try:
            return self._execute(...)
        except Exception as e:
            return self._error_dp(e)   # يستحيل الفشل الصامت بنيوياً
```

- وكيل مدفوع خارج `/deepen` = يستحيل استدعاؤه (لا يعتمد على
  تذكُّر إضافة حارس في api.py)
- استثناء غير متوقع = `DataPoint(None, note=السبب)` تلقائياً
  (البند ٤ مفروض لا مُراجَع)
- **ست ثغرات من تاريخ المشروع كانت ستنحذف من الوجود بهذا
  القرار الواحد.**

**قاعدة التطبيق:** يُبنى `BaseAgent` عند أول وكيل جديد يُضاف
(أو عند دمج أول زوج من ٩.٢) — والوكلاء القائمون يُهاجرون إليه
تدريجياً، وكيلاً مع كل دورة صيانة.

## ٩.٥ ما يُثبَّت كقواعد دائمة (نجحت في V3)

- سقف التكلفة والمصادقة **قبل** أول وكيل، لا بعده
- عمود `outcome` + `outcome_date` في جدول التحليلات من اليوم
  الأول — تتبّع النتائج الفعلية (تكلفته صفر الآن، قيمته سجل
  مصداقية تراكمي لا يملكه منافس)
- مزوّد واحد (Anthropic) — لا fallback تلقائي لمزوّد ثانٍ عند
  بلوغ السقف؛ التوقف المُعلَن هو التصميم الصحيح

# القسم الثالث: تصميم المخرجات الموحّد

> مصدر هذا القسم: دروس مباشرة من عيوب المخرجات المكتشفة
> ("تحليل كلود الأجوف" = خلل ربط عرض، رقم الثقة المختلق،
> أزمة قناة الإرفاق).

## ١٠.١ المبدأ: قالب واحد "أصل" والباقي مشتقات

```
نتيجة التحليل (JSON مكتمل، بخيوط التقاطع إن وُجدت)
        │
   [قالب موحّد واحد — طبقة عرض مشتركة]
        │
   ┌────┼────────┬──────────┬─────────────┐
 لوحة   تقرير    تقرير      JSON خام     (لاحقاً:
 ويب    كامل     مختصر      (تصدير)      PPTX...)
```

**المبرر:** خلل "تحليل كلود الأجوف" كان في ربط العرض لا البيانات.
أربعة مسارات عرض منفصلة = أربع فرص لنفس الخطأ؛ مسار مشترك =
الخطأ يقع مرة ويُصلح مرة.

## ١٠.٢ اللوحة (Dashboard) — المخرج الأول لا الأخير

تُبنى قبل التقارير (عكس ترتيب V3 الحالي) لأنها الوحيدة التفاعلية
والتقارير لقطات ثابتة منها. هرمها من الأعلى:

1. **القرار أولاً:** GO/WATCH/NO-GO بحجم بارز + الثقة المحسوبة
   (X من ٥ مجموعات) + سطر "لماذا" واحد
2. **قسم "موقعك التنافسي"** (من محرّك التقاطع، القسم الأول) —
   يتضمن هامش الربح مقابل منافسين بالاسم
3. **بطاقة لكل مجموعة** قابلة للفتح، حالتها ظاهرة بنفس البروز:
   ✅ كاملة / ⚠️ جزئية / ❌ غير متوفرة — الفجوة معلومة قيّمة
   لا عيب يُخفى
4. **الخريطة داخل بطاقة المنافسة** — وسيلة فهم "من أين يأتي
   المنافسون"، لا تبويب مستقل

## ١٠.٣ التقرير الكامل (Word)

- صفحة الخلاصة التنفيذية أولاً (نجحت باختبار الخمس ثوانٍ — تُثبَّت)
- **سطر مصدر تحت كل رقم** (مصدر + تاريخ سحب + ثقة) مبني في
  القالب نفسه — يستحيل بنيوياً ظهور رقم بلا نسب
- **قسم "حدود هذا التقرير"** قبل التوصيات — يسرد الفجوات صراحة

## ١٠.٤ التقرير المختصر — منتج مختلف لا نسخة مصغرة

صفحة واحدة بتصميم "رسالة جوال" (الواقع: يُقرأ على الجوال أو
مطبوعاً بعجلة): القرار + ٣ أرقام حاسمة + سطرا الموقع التنافسي
(أقرب منافس قابل للمنافسة + أفضل باب دخول) + رابط اللوحة.

## ١٠.٥ إضافة خامسة رخيصة: تصدير JSON خام

زر "حمّل البيانات الخام" — تكلفته صفر (البيانات موجودة)، وقيمته
عالية لشركة تُدخل الأرقام في أنظمتها الداخلية. المنصات التي تحبس
بياناتها بمخرجات مغلقة تخسر ثقة المستخدم المحترف.

## ١٠.٦ قاعدة القبول — درس أزمة الإرفاق

**كل مخرج يُولَّد منه نموذج فعلي ببيانات حقيقية ويُحفَظ في المستودع
نفسه** (`samples/report_full_latest.docx` وأشباهه) مع كل تعديل
على طبقة العرض. المراجع البشري يفتح الملف من المستودع مباشرة —
لا قنوات إرفاق، لا لقطات، لا "أرسلتُها ولم تصل".

## ١٠.٧ ترتيب التطبيق التدريجي للقسم الثالث

1. قاعدة `samples/` (١٠.٦) — فورية، تُطبَّق على مخرجات V3 الحالية
   نفسها (تحلّ أزمة الإرفاق القائمة أصلاً)
2. تصدير JSON الخام (١٠.٥) — رخيص، يُضاف مع أول تعديل على api
3. القالب الموحّد (١٠.١) — يُبنى عند إضافة قسم "موقعك التنافسي"
   (لأن ذاك القسم سيُضاف للمخرجات الثلاثة أصلاً — فرصة التوحيد
   الطبيعية)
4. إعادة ترتيب هرم اللوحة (١٠.٢) — مع القالب الموحّد
5. أسطر المصادر وقسم الحدود (١٠.٣) — مع القالب الموحّد

# القسم الرابع: محرّك اكتشاف الفرص المعكوس (Opportunity Discovery)

> **الفكرة بجملة:** قلب اتجاه السؤال. المنصة اليوم تجيب "عندي
> منتج — أين أبيعه؟". هذا القسم يضيف: **"عندي سوق مستهدف (أو
> قدرة تصدير عامة) — ما المنتجات المطلوبة فيه التي تمثّل فرصة
> حقيقية لمصدّر سعودي؟"** — الاتجاه الأثمن لبيت تصدير مثل سِلك،
> لأنه يكشف فرصاً لم تكن على الرادار أصلاً.
>
> **الترتيب:** يُبنى بعد محرّك التقاطع (القسم الأول) واستقراره —
> لأنه يعيد استخدام مكوناته.

## ١١.١ المدخل

المستخدم يختار **سوقاً** (دولة) بدل منتج، مع مرشّحات اختيارية:

- قطاع عام (غذائي / صناعي / نسيج / الكل)
- حد أدنى لحجم الاستيراد (لتجاهل الأسواق الهامشية)
- تفضيل: "منتجات تصدّرها السعودية فعلاً" أم "أي فرصة حتى لو
  تتطلب توريداً" (وضع بيت التصدير: سِلك تستورد خاماً وتصدّر —
  فالخيار الثاني مشروع لها)

## ١١.٢ منطق الاكتشاف — أربع إشارات من بيانات موجودة أصلاً

**لا وكلاء جدد.** المحرّك يعيد استخدام المصادر القائمة بأسئلة معكوسة:

| الإشارة | المصدر (قائم) | ما تكشفه |
|---|---|---|
| **نمو الاستيراد** | Comtrade — أعلى رموز HS نمواً في استيراد السوق عبر ٣ سنوات | الطلب المتصاعد فعلياً، لا المفترض |
| **فجوة الحصة السعودية** | Comtrade — منتجات يستوردها السوق بكثافة وحصة السعودية فيها < X% رغم كون السعودية مصدّراً عالمياً لها | الفرصة المهدورة تحديداً |
| **القرب اللوجستي المُثبت** | Comtrade — منتجات يستوردها السوق من دول أبعد/أغلى شحناً من السعودية | ميزة تنافسية جغرافية جاهزة |
| **الموسمية القادمة** | trends (pytrends) — طلب موسمي يقترب ذروته في السوق | توقيت الدخول، لا الدخول فقط |

النتيجة: **قائمة مرتّبة بأعلى ١٠-١٥ فرصة** (رمز HS + اسم المنتج +
الإشارات التي رشّحته + قوة كل إشارة)، كل رقم فيها موسوم بمصدره —
نفس مبدأ "لا اختلاق" حرفياً.

## ١١.٣ الربط بمحرّك التقاطع — الدورة الكاملة

القيمة الحقيقية في **الوصل لا العزل**: كل فرصة في القائمة لها زر
"حلّل هذه الفرصة بالكامل" → يشغّل التحليل الاعتيادي (الوكلاء +
التقاطع) على ذلك المنتج × ذلك السوق. الدورة تصير:

```
اكتشاف (ما المطلوب؟) ← تحليل (هل أقدر أنافس؟) ← قرار (GO/NO)
        │                                            │
        └────────── عمود outcome يغذّي ◄─────────────┘
                    ترشيحات المستقبل
```

مع الزمن، عمود `outcome` (القسم ٩.٥) يجعل الترشيحات أذكى:
فرص شبيهة بما نجح فعلياً تتقدم في الترتيب — سجل المصداقية
التراكمي يخدم الاكتشاف أيضاً، لا التحليل فقط.

## ١١.٤ حدود صريحة (ما لا يدّعيه المحرّك)

- **لا يتنبأ بالمستقبل** — يكشف أنماطاً قائمة في بيانات تاريخية
  ويعلنها كذلك ("نما استيراده ٤٠% في ٣ سنوات" لا "سينمو")
- **لا يعرف قدرتك الإنتاجية** ما لم تُدخلها — الترشيح فرصة سوق،
  لا خطة عمل؛ جدواها لك تتحدد بالتحليل الكامل بعده
- **إشارة pytrends تكميلية لا أساسية** — بيانات بحث لا شراء؛
  وزنها في الترتيب أقل من إشارات Comtrade الثلاث
- **قائمة الفرص تُخزَّن بتاريخها** — فرصة اليوم قد تُشبع بعد
  ٦ أشهر؛ إعادة التشغيل الدوري (وكيل المراقبة المقترح سابقاً)
  يحدّثها

## ١١.٥ معايير قبول خاصة بهذا القسم

1. تشغيل الاكتشاف على سوق واحد يُرجع قائمة يكون كل بند فيها
   قابلاً للتتبع: الإشارات المذكورة موجودة حرفياً في استجابات
   Comtrade/trends الخام (تحقق عكسي عيّني على ٣ بنود)
2. منتج بلا أي إشارة حقيقية لا يظهر في القائمة إطلاقاً — لا
   "حشو" للوصول لعدد معيّن؛ إن وُجدت ٤ فرص فقط تُعرض ٤ بصدق
3. زر "حلّل هذه الفرصة" يمرّر رمز HS الصحيح للتحليل الكامل
   دون إعادة إدخال يدوي
4. صفر مصادر بيانات جديدة — يُثبت أن المحرّك يعمل حصراً على
   Comtrade + trends القائمين

# القسم الخامس: ترقية طبقة الامتثال — من بحث إلى قوائم تحقق تنفيذية

## (تركيز: الأسواق صعبة الاشتراطات — أوروبا نموذجاً)

> **الفكرة بجملة:** صعوبة الأسواق المتقدمة (أوروبا تحديداً) ليست
> عائقاً أمام الأتمتة بل **أفضل حالاتها** — لأن نظامها مقنّن بلوائح
> مرقّمة وقوائم منشورة تُراجع بإيقاع معلن، لا نصوصاً متناثرة.
> السوق الأصعب على المصدّر هو الأسهل على المحرّك.
>
> **الترتيب:** يُبنى بعد محرّك التقاطع — قائمة التحقق تتخصص أكثر
> عندما تعرف بطاقة المنتج (الفئة تحدد اللوائح المنطبقة).

## ١٢.١ لماذا أوروبا هي الاختبار الأقسى — والأسهل أتمتةً

المفارقة المعمارية:

| | الخليج | أوروبا |
|---|---|---|
| **طبيعة الاشتراط** | إجراءات (سابر، حلال، تسجيل) موزعة على بوابات | **لوائح مرقّمة** منشورة في EUR-Lex، كل اشتراط قابل للاستشهاد بمادته |
| **مصدر التحديث** | تعاميم وقرارات تصدر بوتيرة سريعة وغير منتظمة | ملاحق تُراجع **بإيقاع معلن** (الرقابة المشددة ~كل ٦ أشهر) |
| **قابلية البناء كمرجع** | خريطة تُجمع يدوياً وتُصان | خريطة **تُشتق من مصدر واحد رسمي** وتُحدَّث بمزامنة الملاحق |

النتيجة: طبقة المرجع الثابت (الطبقة ١) تُبنى لأوروبا **بدقة أعلى**
من أي سوق آخر، والبحث الحي (الطبقة ٢) يستهدف "هل تغيّر الملحق
الفلاني في المراجعة الأخيرة؟" بدل استعلامات عامة.

## ١٢.٢ خريطة الاشتراطات الأوروبية للمنتج الغذائي (سلسلة القرار)

المرجع الثابت يُبنى كسلسلة أسئلة مرتّبة — كل سؤال يحيل للائحته:

1. **هل الدخول مسموح أصلاً؟** المنتجات الحيوانية المصدر تتطلب أن
   تكون الدولة **والمنشأة** على القوائم الأوروبية المعتمدة
   (إطار اللائحتين 2017/625 و2022/2292) — إن لم تكن المنشأة
   السعودية مدرجة، فكل ما بعده بلا معنى. هذا **أول بند** بقائمة
   التحقق لا آخرها.
2. **هل الفئة تحت رقابة مشددة؟** اللائحة 2019/1793 تنشر ملاحق
   بفئات/دول تخضع لنسب فحص مرتفعة أو شهادة رسمية + تقرير مختبر
   **لكل شحنة** — الملاحق تُراجع دورياً (~كل ٦ أشهر)، وهذا هو
   الإيقاع الذي تتزامن معه الطبقة ١.
3. **حدود المتبقيات والملوثات:** متبقيات المبيدات (اللائحة
   396/2005 — قاعدة بيانات MRL قابلة للاستعلام بالمنتج × المادة)
   والملوثات (اللائحة 2023/915). هنا تسقط أغلب الشحنات الغذائية
   عملياً — يستحق تنبيهاً صريحاً بقائمة التحقق: "افحص مخبرياً
   قبل الشحن، لا عند الوصول".
4. **هل المنتج "غذاء جديد"؟** اللائحة 2015/2283 — مكونات لم
   تُستهلك بشكل معتبر في أوروبا قبل 1997 تتطلب ترخيصاً مسبقاً
   (يمس منتجات مبتكرة من مكونات محلية).
5. **الملصق:** اللائحة 1169/2011 — بلغة الدولة المستوردة،
   بمتطلبات إلزامية (مغذيات، مسببات حساسية، منشأ).
6. **الإجراء الحدودي:** إخطار مسبق عبر TRACES (وثيقة CHED)
   بنقطة تفتيش حدودية BCP + مستورد أوروبي مسجّل برقم EORI —
   يعني **وجود شريك استيراد أوروبي شرط تشغيلي لا ورقي**.

> ملاحظة صيانة: أرقام اللوائح أعلاه مستقرة لكن ملاحقها حية —
> التحديث الربع سنوي للطبقة ١ يشمل التحقق من آخر نسخة معدّلة
> (consolidated version) لكل لائحة، ومزامنة ملاحق 2019/1793
> مع كل مراجعة نصف سنوية.

## ١٢.٣ الطبقات الثلاث مطبّقة على أوروبا

- **الطبقة ١ (مرجع ثابت):** سلسلة القرار أعلاه مخزّنة كخريطة:
  فئة HS ← اللوائح المنطبقة ← البنود العملية. تُشتق من EUR-Lex
  وقوائم المفوضية مباشرة — مصدر واحد رسمي، لا تجميع من مدونات.
- **الطبقة ٢ (بحث حي مستهدف):** الوكيل لا يسأل "EU import
  requirements dates" بل يسأل **أسئلة تحقق محددة**: "آخر مراجعة
  لملاحق 2019/1793 — هل فئة المنتج/السعودية مدرجة؟"، "آخر تعديل
  MRL للمادة الفعالة X على المنتج Y". البحث يتحقق من نقاط معلومة
  بدل أن يكتشف من الصفر.
- **الطبقة ٣ (قائمة التحقق التنفيذية):** بنفس بنية قائمة الخليج
  لكن مرتّبة بسلسلة القرار الأوروبية: الأهلية أولاً (بند ١) ←
  الفحوص المخبرية قبل الشحن (بند ٣) ← الشريك المستورد وTRACES
  (بند ٦) — مع وسم كل بند بمرجعه القانوني ودرجة ثقته.

**قرار معتمد — صفر وكلاء جدد:** الطبقة ١ ليست وكيلاً بل بيانات
ثابتة في SQLite (الهجرة إلى Postgres قرار مؤجل عند الحاجة)
تُزامَن بسكربت خارجي (خارج مسار التحليل،
نفس فلسفة `correlation.py`)؛ الطبقة ٢ ترقية لوكيل الاشتراطات
**القائم** ضمن الـ21 (يقرأ الخريطة من الذاكرة ثم يبحث أسئلة
تحقق فقط)؛ الطبقة ٣ عمل قالب عرض (القسم الثالث). اتجاه الخروج
السعودي فرع ثانٍ داخل نفس الوكيل، لا وكيل مستقل. وعند بناء
`BaseAgent` (٩.٤) يكون هذا الوكيل أول مرشح للهجرة إليه.

## ١٢.٤ مصادر الطبقة ١ الجاهزة — لا بناء من الصفر

منصات دولية مجانية تغطي كل الأسواق (لا أوروبا فقط) تُعتمد
كمغذّيات أولية للمرجع الثابت:

| المنصة | الجهة | ما تقدمه للطبقة ١ |
|---|---|---|
| **Market Access Map** (macmap.org) | ITC | التدابير غير الجمركية (NTMs) والاشتراطات التنظيمية والتعرفات لأي منتج × أي سوق — **تنزيل بالجملة مجاني** = المغذّي الأساسي للخريطة |
| **Global Trade Helpdesk** (globaltradehelpdesk.org) | ITC + WTO + UNCTAD | بحث موحّد HS × دولة يجمع بيانات 11 منظمة دولية (اشتراطات، تعرفات، إحصاءات) — مصدر تحقق ثانٍ |
| **Access2Markets** (المفوضية الأوروبية) | EU | قواعد واشتراطات المنتجات للاتحاد الأوروبي + أكثر من 120 سوقاً — المرجع الأدق للمسار الأوروبي (قسم ١٢.٢) |

**قواعد الاعتماد:**

1. هذه المنصات **مغذّيات لا بدائل** — تعطي قائمة لوائح خام
   بالإنجليزية؛ التحويل لقائمة تحقق عربية تنفيذية مرتّبة
   بالتسلسل والتكلفة هو عمل المحرّك (وميزة سِلك)
2. كل بند مستورد منها يُوسم بمصدره وتاريخ سحبه (نفس مبدأ
   شفافية المصدر — `source: macmap` كوسم explee بالضبط)
3. التحديث الربع سنوي للطبقة ١ = إعادة مزامنة مع هذه المصادر،
   لا إعادة بحث حر
4. **فجواتها الأربع تبقى ملك المحرّك:** لا قوائم تنفيذية،
   لا عربية، لا ربط ببطاقة المنتج، لا اتجاه خروج سعودي —
   هذه الأربعة هي القيمة المضافة التي لا تُستورد

## ١٢.٥ التعميم — تصنيف الأسواق حسب "قابلية التقنين"

أوروبا ليست حالة خاصة بل **الطرف الأعلى في طيف**، والمحرّك يعلن
موقع كل سوق عليه بشفافية (نفس مبدأ "الفجوة معلومة قيّمة"):

| الدرجة | أمثلة | سلوك المحرّك |
|---|---|---|
| **مقنّن بالكامل** | الاتحاد الأوروبي، بريطانيا | مرجع مشتق من لوائح مرقّمة، بحث حي للتغييرات فقط — ثقة عالية |
| **شبه موحّد** | الخليج (سابر/SFDA/حلال) | خريطة مبنية يدوياً من مصادر رسمية، تحديث ربع سنوي — ثقة متوسطة-عالية |
| **موثّق جزئياً** | أسواق عربية وآسيوية | مرجع أساسي + اعتماد أكبر على البحث الحي — قائمة التحقق تحمل وسم "تحقق محلياً" على البنود غير المؤكدة |

القاعدة الصارمة: **درجة الثقة تظهر على قائمة التحقق نفسها** —
بند مصدره لائحة أوروبية مرقّمة ≠ بند مصدره نتيجة بحث عن سوق
قليل التوثيق، والمستخدم يرى الفرق.

## ١٢.٦ اتجاه الخروج السعودي (يبقى كما اعتُمد)

قائمة تحقق الدخول الأوروبية تُعرض جنب **قائمة تحقق الخروج
السعودية** (تصاريح SFDA للفئات الخاضعة، شهادة المنشأ، متطلبات
هيئة الصادرات) — الاتجاهان معاً هما الخدمة التي لا يقدمها منافس
أجنبي، وتزداد قيمتها كلما صعُب السوق المستهدف: المصنع الذي يستهدف
أوروبا يحتاج الاتجاهين متزامنين (اعتماد المنشأة أوروبياً يمر عبر
الجهة السعودية المختصة أصلاً).

## ١٢.٧ معايير القبول الخاصة بهذا القسم

1. تحليل منتج غذائي × سوق أوروبي يُخرج قائمة تحقق كل بند فيها
   موسوم بمرجعه (رقم لائحة أو مصدر رسمي) — صفر بنود بلا نسب
2. اختبار "الأهلية أولاً": منتج حيواني المصدر من منشأة غير
   مدرجة أوروبياً → القائمة تبدأ بتحذير الأهلية ولا تسرد البنود
   التالية كأن الطريق سالك
3. اختبار المزامنة: بعد مراجعة نصف سنوية لملاحق 2019/1793،
   الطبقة ١ تعكس التغيير خلال دورة التحديث التالية (تحقق يدوي
   عيّني على فئة واحدة)
4. سوق "موثّق جزئياً" يُخرج قائمة بوسوم "تحقق محلياً" الصريحة —
   لا يُعرض بنفس ثقة السوق المقنّن (اختبار عكسي: نفس المنتج على
   ألمانيا ثم على سوق أفريقي — درجتا الثقة تختلفان ظاهرياً)
5. قائمتا الدخول والخروج تظهران معاً في التقرير الكامل واللوحة،
   وسطر واحد بالمختصر
6. كل بند مصدره منصة دولية (macmap/GTH/Access2Markets) يحمل
   وسم المصدر وتاريخ السحب — تحقق عيّني على ٣ بنود بأنها موجودة
   حرفياً في المصدر المذكور
